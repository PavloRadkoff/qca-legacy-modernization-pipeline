# Whitepaper 21: The Network Nervous System
### Мережева архітектура як проект: Від ізоляції Docker-контейнерів до залізного щита Nginx та Firewall

**Автор:** Pavlo Radkoff | Systems Engineer & QCA Architect  
**Домен:** Network Engineering, Security Hardening, Docker Networking, Reverse Proxy  

---

## 1. Парадокс "Плоскої Мережі" (The Flat Network Trap)

Найбільша помилка при розгортанні серверів — це створення "плоскої" структури, де всі сервіси бачать один одного, а база даних доступна з будь-якої точки локальної мережі. У такій системі злам одного другорядного скрипта означає миттєвий компроміс усієї інфраструктури, включаючи банківське ядро чи медичні записи.

В інженерному проекті **QCA.Forge** мережа — це не просто кабелі та IP-адреси. Це **ешелонована нейросистема**, де кожен сегмент має свій рівень доступу та "стерильності".

## 2. Мікро-сегментація: Docker Networks та Ізоляція

Ми ніколи не покладаємося на стандартний міст `docker0`. Кожен проект (наприклад, Медична АІС) отримує власну ізольовану мережу (Software Defined Network):

* **Back-end Network:** Закрита мережа без доступу до інтернету. Тут живуть контейнери з логікою на C# та базами даних PostgreSQL. Вони не знають про існування зовнішнього світу, що унеможливлює витік даних через бекдори.
* **Front-end Network:** Вузький місток, через який веб-інтерфейс спілкується з бекендом.
* **Database Isolation:** База даних ніколи не "публікує" порти (no ports mapping). Вона доступна лише за внутрішнім DNS-іменем контейнера і лише для тих сервісів, які мають на це авторизоване право.

## 3. Nginx: Електронний Гейткіпер (The Gatekeeper)

Nginx у нашій архітектурі — це не просто веб-сервер. Це **Shield (Щит)**. Він є єдиною точкою входу (`Ingress`) у систему:

1. **SSL Termination:** Весь трафік шифрується (TLS 1.3). Внутрішні мережі працюють на високій швидкості без шифрування, але зовні сервер виглядає як неприступна фортеця.
2. **Reverse Proxy & Hardening:** Nginx приховує реальну топологію серверів. Він обрізає підозрілі HTTP-заголовки, лімітує кількість запитів (Rate Limiting), запобігаючи Brute-force та DoS-атакам на рівні прикладного софту.
3. **Security Headers:** Ми примусово впроваджуємо політики HSTS, Content Security Policy (CSP) та X-Frame-Options, захищаючи користувачів від XSS та Clickjacking.

## 4. Firewall (UFW/IPTables): Залізна Завіса

Якщо Nginx — це охоронець на вході, то Firewall — це зачинені броньовані двері. 
Мій принцип: **Заборонено все, що не дозволено явно.** На системному рівні Ubuntu LTS ми налаштовуємо суворі правила:
* Закриті всі порти, крім 80/443 для трафіку та кастомного порту для SSH з обмеженням по IP.
* **ICMP Rate Limiting:** Захист від "пінг-штормів".
* **Connection Tracking:** Відстеження станів з'єднань, щоб сервер не відповідав на запити, які він не ініціював.

## 5. Оркестрація Мережевого Хаосу через ШІ

Проектування мережі — це сотні рядків конфігурацій `nginx.conf`, `docker-compose.yml` та правил `iptables`. Помилка в одному символі може відкрити дірку в безпеці.

Мій ШІ-екзоскелет ("Шаманік") виступає аудитором мережевої топології. Я задаю інтент:
*"Спроектуй мережу для Worker Node 1. Потрібна повна ізоляція БД PostgreSQL від інтернету. Доступ до БД має мати лише контейнер `ais-api`. Весь зовнішній трафік через Nginx з лімітом 10 запитів/сек для одного IP. Згенеруй Docker-network маніфести та конфіг Nginx"*.

ШІ видає технічно досконалий код, який я верифікую. Це дозволяє мені, як Архітектору, тримати в голові складну топологію десятків вузлів, не тонучи в рутині мережевого адміністрування.

### Висновок

Мережа в **QCA.Forge** — це не сервіс, це **Процес Безпеки**. Ми будуємо інфраструктуру так, щоб навіть у разі зламу одного компонента, вся інша система залишалася холодною та недосяжною для атакуючого. Це і є справжня інженерна надійність.
