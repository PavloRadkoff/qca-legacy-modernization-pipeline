# Whitepaper 19: The Dirty Data Paradox 
### Чому строгий C# ламається на "брудних" даних, і як старий добрий Python рятує Enterprise-міграції

**Автор:** Pavlo Radkoff | Systems Engineer & QCA Architect  
**Домен:** Data Engineering, ETL Pipelines, Pragmatic Downgrade, Type Systems  

---

## 1. Ілюзія Ідеального Світу (The Clean Room Fallacy)

Сучасні Enterprise-фреймворки (як-от C# .NET 8) створювалися для "стерильного" світу. Вони вимагають строгих контрактів, чітко визначених типів даних (Int, Decimal, DateTime) та передбачуваних REST API. 

Але коли ми переводимо реальний бізнес зі старої 1С або розрізнених філій на нову платформу, ми стикаємося з реальністю. Реальні корпоративні дані — це хаос. Це Excel-таблиці з тисячами рядків, де в колонці `Кількість` (яка має бути числом) раптом зустрічається текст *"немає на складі"*, а в колонці `Дата народження пацієнта` написано *"десь у травні"*.

## 2. Чому "модний" C# тут капітулює

Якщо ви спробуєте імпортувати цей сирий хаос безпосередньо у строгий C# бекенд, система капітулює на першому ж рядку. Серіалізатор (наприклад, Entity Framework або CsvHelper) спробує перетворити слово *"немає"* на `System.Decimal` і миттєво викине фатальну помилку (`FormatException` або `SerializationException`).

Щоб змусити C# обробити такі дані, програмісту доведеться написати сотні рядків "костилів" (Custom Binders, гігантські блоки `try-catch`, складні перевірки регулярними виразами для кожного поля). Чистий код перетворюється на нечитабельне сміття. Строгість C#, яка є його головною суперсилою в транзакціях, тут стає його головним недоліком.

## 3. Прагматичний Даунгрейд у дії: Python як "М'ясорубка"

Для вирішення проблеми "брудних даних" ми застосовуємо нашу парадигму Прагматичного Даунгрейду. Ми свідомо відмовляємося від C# на етапі первинного збору (Ingestion) і беремо до рук старий добрий Python.

**Чому Python? Тому що його динамічна типізація ("Duck Typing") ідеально підходить для хаосу.**

Інструменти на кшталт бібліотеки `pandas` у Python створені для того, щоб "проковтнути" що завгодно. 
1. **Динамічний парсинг:** Python завантажить Excel-файл з помилками, не зупиняючи виконання програми. Замість "падіння", він просто помітить аномальні поля як `NaN` (Not a Number).
2. **Векторна очистка:** Те, що в C# вимагало б 50 рядків циклів та перевірок `if/else`, у Python робиться одним рядком: *знайти всі нечислові значення в колонці "Ціна", замінити їх на нуль, а пропущені дати заповнити стандартним значенням*.
3. **Нормалізація форматів:** Python легко "витягує" логіку з текстового сміття (наприклад, забирає зайві пробіли, кириличні символи замість латинських, форматує телефони) завдяки потужним скриптовим рушіям.

## 4. Патерн "Пральна Машина Даних" (Data Washing Machine)

Ми не пишемо бізнес-логіку на Python. Ми використовуємо його як ізольований мікросервіс — "Пральну машину". 

**Архітектура процесу виглядає так:**
1. **Ingestion (Python):** Python-скрипт забирає брудні Excel/DBF файли з легасі-системи. 
2. **Cleaning (Python):** Дані нормалізуються, сміття відфільтровується, типи приводяться до єдиного формату.
3. **Strict Export:** Python формує ідеально чистий, строго типізований JSON (або тимчасову SQL-таблицю).
4. **Core Processing (C# .NET 8):** Наш новий, стерильний C#-бекенд спокійно "всмоктує" ці ідеальні дані без жодної помилки, розкладаючи їх по реляційних таблицях і застосовуючи фінансову математику.

### Висновок

Інженерний прагматизм полягає у виборі правильного інструменту для кожної фази. Змушувати C# парсити брудні Excel-файли — це як забивати цвяхи мікроскопом. Набагато ефективніше делегувати "брудну роботу" гнучкому Python, залишивши для C# те, що він робить найкраще: транзакційну безпеку, швидкість і строгу бізнес-логіку підприємства.
